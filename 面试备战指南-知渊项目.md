## 一、30秒电梯演讲（面试官说"介绍一下这个项目"）

**标准版（约45秒）：**

> 知渊是我从0到1独立开发的一个 Agentic RAG 智能知识库平台。它跟普通RAG最大的区别是——我让 Agent 做了"检索质量裁判"：Agent 自己决定什么时候检索、检索什么、检索结果好不好、不好怎么补救。
>
> 具体来说，用户提问后，Agent 自主调度混合检索——稠密向量加自实现的BM25双路召回，RRF融合后用Jina重排序，再做两阶段自动父块合并。然后 LangGraph 管道里的一个小模型来给检索结果打分——达标直接回答，不达标自动选三种策略重写查询再搜。
>
> 前端把 Agent 每一步在想什么、检索到了什么、评分多少全部实时可视化出来。整个系统 FastAPI异步后端加SSE流式、JWT鉴权、Docker Compose五行命令部署。

**三句话精简版：**

> 我做了一个让 Agent 自己判断检索质量并自适应补救的 RAG 平台。核心技术栈是 LangChain Agent + LangGraph 状态图 + Milvus 混合检索。全栈自研，从文档解析到前端可视化完整闭环。

## 三、核心技术逐个深挖（附面试回答模板）

### 3.1 Agent 架构

**你的代码实际做了什么：**
- 用 LangChain `create_agent()` 创建 ReAct Agent
- LLM 是 Qwen3-Max（通过 OpenAI 兼容端点，`temperature=0.3`）
- 注册了3个工具：`search_knowledge_base`、`search_web`、`get_current_weather`
- `recursion_limit=8`（最多8步推理）
- 每轮只允许调用1次知识库检索（防死循环）
- 超过50条消息自动摘要压缩前40条

**面试官问："为什么用 ReAct 而不是其他 Agent 模式？"**

> ReAct 模式天然适合 RAG 场景——Agent 需要先"思考"要不要检索、检索什么，然后"行动"去检索，拿到结果后再"观察"质量决定要不要继续。这是一个典型的 Think→Act→Observe 循环。相比之下 Plan-and-Execute 更适合多步任务规划，对我的场景来说过重了。

**面试官追问："你做了什么防护措施？"**

> 三个防护：第一，`recursion_limit=8` 限制最大推理步数；第二，每轮最多调用一次知识库检索，防止 Agent 反复搜同一个东西；第三，系统提示明确要求拿到工具结果后必须立即回答，禁止工具链式调用。这些是生产环境必须做的——LangChain Agent 不加限制很容易陷入死循环。

**面试官问："对话上下文怎么管理的？"**

> 用 PostgreSQL 做持久化存储，Redis 做热缓存（TTL 300秒）。关键设计是超过50条消息时自动触发摘要压缩——把前40条压缩成一段摘要作为 SystemMessage，只保留最近10条完整消息。这样既保住了长对话的语义连贯性，又不超出 LLM 的上下文窗口。摘要是用同一 LLM 生成的一句话总结。

### 3.2 LangGraph 多阶段 RAG 管道

**你的代码实际做了什么：**

```
retrieve_initial ──→ grade_documents ──┬── related ──→ END (生成回答)
                                       │
                                       └── not related ──→ rewrite_question
                                                                │
                                                                ▼
                                                       retrieve_expanded ──→ END
```

**四个节点：**

| 节点 | 做什么 | 用了什么模型 |
|------|--------|-------------|
| `retrieve_initial` | 混合检索 top_k=5，候选池扩大到15 | BGE-M3 + BM25 (本地) |
| `grade_documents` | 评估检索结果相关性，二分判断 yes/no | Qwen3.6-Flash (小模型，temperature=0) |
| `rewrite_question` | 从3种策略中选一种重写查询 | Qwen3-Max (主模型，temperature=0) |
| `retrieve_expanded` | 按策略执行扩展检索（最多2路），去重合并 | 同 retrieve_initial |

**条件边逻辑：**
- 评分模型不可用 → 默认走重写（宁可多搜一次，不错过）
- 结构化输出失败 → 默认为相关（不阻塞用户）
- 扩展检索后多路结果去重（按 `filename + page_number + text` 三元组）

**面试官问："为什么用 LangGraph 而不是自己写 if-else？"**

> LangGraph 给了我有状态的图执行引擎。RAG 检索不是线性流程——检索完要评分，评分决定走哪条分支，扩展检索完要聚合结果。用状态图把这些节点和条件边建模出来，逻辑清晰可测试，而且 LangGraph 天然支持每个节点独立调试。如果自己写 if-else，分支一多就成一团乱麻。

**面试官问："三种查询重写策略怎么选的？"**

> 通过结构化输出让 LLM 自动选。我在 prompt 里给了明确的选择标准：包含具体名称/代码/数字的走 Step-back（先理解通用概念再回来），模糊概念性问题走 HyDE（生成假设文档去匹配），多步骤复杂问题走 Complex（两种都用）。选型模型用 Qwen3.6-Flash，成本低、速度快。如果结构化输出失败，默认降级为 Step-back。

**面试官追问："Step-back 具体怎么实现？"**

> 两步走。第一步，让 LLM 把用户的具体问题抽象成更高层次的"退步问题"——比如用户问"为什么Redis用跳表不用B+树"，退步问题就是"跳表和B+树的原理和适用场景分别是什么"。第二步，让 LLM 回答这个退步问题，把结果拼接到扩展查询里一起检索。这样检索到的文档是通用原理级别的，能补上原问题可能漏掉的背景知识。

### 3.3 混合检索

**你的代码实际做了什么：**

**稠密向量这条线：**
- 模型：`BAAI/bge-m3`，1024维，内积相似度（IP）
- 本地 CPU 推理，`normalize_embeddings=True`
- Milvus 索引：HNSW（M=16, efConstruction=256），搜索参数 ef=64

**稀疏向量这条线：**
- 完全自己实现的 BM25（`k1=1.5`, `b=0.75`）
- 分词策略：中文逐字切分，英文按词切分
- 词表/文档频率持久化到 `data/bm25_state.json`
- 增量更新：添加/删除文档时不必重建词表
- Milvus 索引：SPARSE_INVERTED_INDEX（drop_ratio_build=0.2）

**融合这条线：**
- 两个 `AnnSearchRequest` 并行发出
- 每个搜 `top_k * 2` 个候选
- PyMilvus `RRFRanker(k=60)` 融合排序
- 最终取 `top_k` 个结果

**降级策略：**
- 混合检索失败 → 自动回退纯稠密检索
- 纯稠密也失败 → 返回空结果

**重排序：**
- Jina Reranker v3（`jina-reranker-v3`），超时15秒
- RRF排名作为输入排序，重排序后按 `relevance_score` 重排
- 重排序失败 → 优雅回退到 RRF 原始排序

**面试官问："为什么BM25要自己实现，不用 Milvus 内置的？"**

> Milvus 2.4+ 内置了 BM25 分词器，但主要面向英文。中文场景需要特定的分词策略——中文逐字（避免分词错误导致召回失败），英文按词。而且自实现让我能完全控制增量更新逻辑——文档增删时只更新词频统计，不重建整个词表索引。词表持久化到 JSON 文件，重启服务不丢失。

**面试官追问："BM25 增量更新怎么做到不回收词表索引的？"**

> 这是有意为之。回收索引意味着词表维度变化，而 Milvus 里已经写入的稀疏向量维度是固定的——如果回收索引，新写入的向量维度跟旧的不一致，检索会出错。所以我选择了"只增不减"的策略：删除文档时减少文档频率计数但不删除词表条目，让词表索引永远单调递增。代价是词表会越来越大，但实际场景下文档量级可控。

**面试官问："RRF 融合的原理是什么？k=60 怎么选的？"**

> RRF 的核心思想是不看分数的绝对值（稠密和稀疏的分数分布差异很大），只看排名。公式是 `score = Σ 1/(k + rank_i)`。k=60 是经验值——太大了不同排名的权重差异变小，太小了排名靠后的文档几乎没权重。这个值在 PyMilvus 文档里有推荐，我在代码里留了注释说可调节。
>
> 通俗理解：假设文档A在稠密检索排第1、稀疏检索排第10，文档B在两边都排第4。RRF 算出 A 的融合分 = 1/(60+1) + 1/(60+10) ≈ 0.0164 + 0.0143 = 0.0307，B 的融合分 = 1/(60+4) + 1/(60+4) = 0.0312。B 得分更高——因为 B 在两个维度的排名更均衡。

### 3.4 三级分块 + Auto-merging

**你的代码实际做了什么：**

| 级别 | 块大小 | 重叠 | 存储位置 |
|------|--------|------|---------|
| L1（根） | 1200字符 | 240 | PostgreSQL `ParentChunk` 表 |
| L2（父） | 600字符 | 120 | PostgreSQL `ParentChunk` 表 |
| L3（叶） | 300字符 | 60 | Milvus（向量化） |

**父子关系：**
- L1 → `parent_chunk_id=""`（无父），`root_chunk_id=自己`
- L2 → `parent_chunk_id=L1的id`，`root_chunk_id=L1的id`
- L3 → `parent_chunk_id=L2的id`，`root_chunk_id=L1的id`
- 只有 L3 叶子层写入 Milvus 做向量检索

**Auto-merging 逻辑：**
- 阈值 `AUTO_MERGE_THRESHOLD=2`
- 两阶段合并：L3→L2，然后 L2→L1
- 同一父块下 ≥2 个叶子命中 → 从 PostgreSQL 取出父块替换这些叶子
- 保留第一个命中的最高分数
- 合并后按分数降序截断到 `top_k`

**面试官问："为什么不直接把大块存入向量库？"**

> 向量检索的本质是语义相似度匹配——块越大，包含的语义越杂，跟查询的相似度就越"稀释"。小块（300字符）能精确匹配到相关段落，但缺点是只拿到片段，丢失了上下文。所以我用小叶子做检索入口，大父块做上下文还原——检索用细粒度，回答用粗粒度。

**面试官追问："阈值为什么是2？"**

> 阈值=2 意思是同一父块下至少有2个叶子被检索命中才触发合并。如果只有1个叶子命中，可能是偶然命中了一个无关片段，贸然展开整个父块反而引入噪音。2个及以上命中说明这一整段跟查询强相关，展开才有价值。这个阈值可以根据知识库类型调节——技术文档可以设低一点，长篇小说可以设高一点。

**面试官问："分块器为什么用 RecursiveCharacterTextSplitter 而不是 SemanticChunker？"**

> 分离器选择是工程权衡。RecursiveCharacterTextSplitter 基于分隔符优先级（`\n\n → \n → 。→ ！→ ？→ ，`）层层回退切分，中文语境下`。`和`！`和`？`是天然的语义边界。SemanticChunker 需要额外做一次 embedding 计算边界相似度，成本和延迟都高。在这个项目里，结合三层分块+自动合并，RecursiveCharacterTextSplitter 的效果已经足够好。

### 3.5 SSE 流式 + 实时可视化

**你的代码实际做了什么：**

流式架构的核心是一个**统一输出队列 + 后台任务**模式：

1. 主协程创建一个 `asyncio.Queue()` 作为事件总线
2. 后台 `_agent_worker` 协程运行 `agent.astream()`，产出 `AIMessageChunk` 推入队列
3. 主循环从队列取事件，`yield` 为 SSE 格式
4. RAG 检索步骤通过 `call_soon_threadsafe` 跨线程安全推入同一个队列（因为 LangChain 工具在线程池中运行）

**事件类型：**
- `content` —— LLM 回答流式 token
- `rag_step` —— 检索过程中的每一步（图标+标签+详情）
- `trace` —— 完整检索追溯数据（Agent 完成后发送）
- `web_sources` —— 网络搜索来源（Agent 完成后发送）
- `error` —— 错误信息
- `[DONE]` —— 结束哨兵

**断开处理：**
客户端断开时（`GeneratorExit`），取消后台 agent 任务，防止资源泄漏。

**面试官问："SSE 流式中最大的技术挑战是什么？"**

> RAG 检索步骤和 LLM 回答 token 的时序协调。按 LangChain 的 `astream()` 默认行为，工具执行期间不会产出任何 chunk——用户看到的就是长时间空白然后突然冒出回答。我的方案是用统一事件队列解耦：检索步骤通过 `call_soon_threadsafe` 实时入队，不等 agent 产出 chunk。这样前端能在检索进行中就展示"正在检索→正在评分→正在重写"，而不是干等。

**面试官追问："call_soon_threadsafe 是什么？为什么需要它？"**

> LangChain 的工具函数在线程池中执行（不在主事件循环中），而 `asyncio.Queue.put_nowait` 不是线程安全的。`call_soon_threadsafe` 把 put 操作调度回主事件循环执行，既保证了线程安全，又保留了实时性。这是 asyncio 中处理跨线程通信的标准做法。

## 四、10个高频深挖问题（附答案）

### Q1："你这个项目的难点在哪里？"

> **三个难点。**
>
> 第一个是**检索质量的自适应闭环**。不是搜一次就完事，而是让 LLM 当裁判——评估检索结果好不好，不好就自动重写查询再搜。这里面的挑战在于：评分模型要够轻量（不能用主模型评，又慢又贵），选策略要够准（三种策略各有适用场景），整个流程不能超时。我的方案是用 Qwen3.6-Flash 做评分（比主模型便宜一个数量级），结构化输出做策略选择，整个管道加了降级保护（评分模型挂了直接走重写）。
>
> 第二个是**BM25 的自实现和增量更新**。直接用 Milvus 内置 BM25 对中文支持不好，自己写的话词表持久化、增量更新、跟 Milvus 稀疏向量的维度对齐都是坑。我的方案是词表只增不减——文档删了不回收索引，避免维度不匹配。代价是词表会增长，但在文档量可控的范围内完全够用。
>
> 第三个是**SSE 流式中的实时步骤可视化**。LangChain 默认的工具执行不产出流式事件，用户只能干等。我用 asyncio 统一事件队列 + call_soon_threadsafe 跨线程调度解决了这个问题。

### Q2："如果给你更多时间，你会优化哪些地方？"

> 三个方向：
> 1. **检索质量评估从二元判断升级为多维度评分**——不只是相关/不相关，而是从准确性、完整性、时效性三个维度打分。目前二元判断太粗暴，有些文档部分相关但不够好，二元判断可能误判为不相关。
> 2. **加一个查询意图分类前置**——事实查询走精确检索、概念解释走 HyDE、对比分析走 Complex。现在全靠 LLM 在评分阶段之后选策略，可以前移并缓存意图分类结果。
> 3. **接入本地部署的重排序模型**（如 BGE-Reranker），替代 Jina API 调用——减少网络延迟和外部依赖，并支持更大候选池。

### Q3："你的系统能支持多大的知识库？"

> 目前是单机部署，受限因素主要是：BGE-M3 的 embedding 推理速度（CPU 上约为每秒10-20条）、Milvus 单节点的向量容量（百万级轻松支持）、BM25 词表的增长。如果要扩展到千万级文档，需要做三件事：BGE-M3 切 GPU 推理、Milvus 分片部署、BM25 换 Elasticsearch。

### Q4："为什么用 LangChain 而不是自己调 API？"

> LangChain 在这个项目里提供了三个关键能力：第一，Agent 框架——`create_agent` 封装了 ReAct 循环、工具调用解析、系统提示注入，自己写至少要500行且容易出 bug。第二，工具装饰器 `@tool` ——给函数加上 description，Agent 自动理解什么时候该调用。第三，LangGraph 状态图——把检索管道建模为可组合的节点图，条件边让决策逻辑可视化。但如果只是调 embedding 或 LLM，我倾向于直接调 API——LangChain 的抽象层在简单场景下反而是负担。

### Q5："Agent 工具调用出错了怎么处理？"

> 代码里做了三层防护：第一层是系统提示约束——禁止重复调用同一工具、拿到结果必须立即回答；第二层是运行时限制——`recursion_limit=8` 硬性限制推理步数；第三层是工具内部降级——混合检索失败自动回退纯稠密检索，Jina Reranker 超时或报错也回退到 RRF 原始排序。最终如果所有检索都失败，Agent 会诚实地说"不知道"而不是编造答案。

### Q6："你为什么要做前端可视化 Agent 思考链？"

> RAG 系统最大的问题是"黑盒"——用户看到答案但不知道信息来源、不知道系统做了什么判断。我把检索模式、评分结果、重写策略、重排序分数、网络来源全部实时展示出来，有两个目的：一是让用户建立信任（答案有据可查），二是方便调试（我开发时一眼就能看出哪一步出了问题）。

### Q7："你的数据库 schema 设计有什么考量？"

> 三个表。`users` 存用户信息和角色，`chat_sessions` 存会话元信息和标题，`chat_messages` 存每一条消息（包括 rag_trace 和 web_sources 的 JSON）。设计要点：消息和会话是外键关联的，标题通过 LLM 自动从用户第一条消息生成（限制15字）。Redis 做读缓存，TTL 300秒，写操作（保存消息）同步更新缓存，删操作（删除会话）同步清除缓存。

### Q8："三个 LLM 分别是什么角色？为什么这么分工？"

> | 模型 | 角色 | 为什么 |
> |------|------|--------|
> | Qwen3-Max | Agent 大脑 + 策略选择 | 需要强推理能力，Temperature=0.3给一点创造性 |
> | Qwen3.6-Flash | 文档相关性评分 | 只做二元判断，轻量模型足够，成本低延迟小 |
> | Qwen3-Max | Step-back/HyDE 生成 | 需要生成高质量的退步问题和假设文档 |

> 核心原则：高频低成本操作用小模型，低频高价值操作用大模型。评分每轮必做，用 Flash 省钱；策略选择和文本生成每轮最多一次，用 Max 保质量。

### Q9："你的项目最有技术含量的设计决策是什么？"

> 我觉得是 **候选池放大→逐级收敛** 的检索策略。具体来说：Milvus 每个通道搜 `top_k * 2` 个候选（双路共约20个），RRF 融合后保留 top_k 个，在 `retrieve_documents` 里放大到 `candidate_k = top_k * 3 = 15` 做 Jina 重排序，重排序后再做 Auto-merging 合并父块，最后截断回 `top_k = 5`。每一级都在缩小范围但每一级都有足够的候选池让下一级精排。

### Q10："让你把这个项目部署到生产环境，还缺什么？"

> 1. 加请求限流（当前没有 rate limiting）
> 2. 加可观测性（Prometheus metrics + 结构化日志）
> 3. Milvus 做集群部署（当前单节点）
> 4. CI/CD 管道（当前是手动部署）
> 5. 加了监控和告警（尤其是 Milvus 连接断开、embedding 模型 OOM）

## 五、面试中"讲故事"的技巧

### 5.1 用"做过什么→为什么这样做→得到了什么结果"三段式回答

比如面试官问"讲讲你的检索系统"：

> **做过什么**：我构建了一个稠密+稀疏的混合检索系统，BGE-M3 做语义匹配，自实现的 BM25 做关键词匹配，通过 RRF 算法融合两路结果，再经 Jina Reranker 精排。
>
> **为什么这样做**：纯向量检索对专有名词、代码片段、数字这类精确匹配效果很差——比如搜"RFC 7231"，向量可能返回 HTTP 相关的文档但不一定精确命中这个标准编号。BM25 天然擅长这种精确词匹配，两者互补。
>
> **什么结果**：最终命中率相比纯向量检索有明显提升。更有意思的是，在混合检索失败时系统会自动降级到纯稠密检索——降级策略保证了可用性不降级。

### 5.2 数字是你的武器

面试中说数字让回答立刻变专业。你的项目中可以用的数字：
- 1024维 BGE-M3 向量、HNSW M=16 efConstruction=256
- BM25 k1=1.5 b=0.75
- RRF k=60
- 三级分块 300/600/1200 字符
- 候选池 top_k×3=15
- JWT 31万轮 PBKDF2
- 50条消息触发摘要压缩
- 5个 Docker 服务一键部署

**用法示例**："我不是简单地把检索结果扔给LLM，而是做了三级逐级收敛——双路各取10个候选，RRF融合到5个，放大到15个做重排序，合并父块后再截断到5个——确保每一级都有丰富的候选池支撑下一级精排。"

### 5.3 展现技术判断力

面试官喜欢听到"我选了A而不是B，因为……"。准备几个这样的对比：

| 你的选择 | 替代方案 | 为什么你的更好 |
|----------|---------|---------------|
| Milvus | Pinecone/Qdrant | 原生支持混合检索+稀疏向量，自托管 |
| 自实现BM25 | Milvus内置/Elasticsearch | 中文分词可控，增量更新灵活 |
| LangGraph | 自写if-else | 状态图可视化，条件边可测试 |
| SSE | WebSocket | 单向数据流够用，实现简单 |
| RecursiveCharacterTextSplitter | SemanticChunker | 中文标点天然语义边界，无额外embedding成本 |

## 七、几个"坑"——千万别这样回答

1. **"我就是照着教程做的"** → 换成："我参考了 LangChain 官方文档的 Agent 模式，然后针对 RAG 场景做了定制——加了工具调用限制、SSE流式适配、检索步骤可视化。"

2. **"这个我也不太懂"** → 换成："这个模块的核心思路是……具体细节我可以展开讲讲我的理解。"（展示思维过程比答案本身更重要）

3. **"就用了这些技术"** → 换成："我选了 X 而不是 Y，因为 X 在 Z 场景下更好。比如……"

4. **回答太短** → 每个核心技术至少讲30秒。技术点回答的黄金结构：**定义→实现→动机→效果**。
